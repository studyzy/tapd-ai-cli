# 功能规范: 提取 tapd-sdk 独立库

**功能分支**: `002-extract-tapd-sdk`
**创建时间**: 2026-04-17
**状态**: 草稿
**输入**: 用户描述: "我希望对internal/client和internal/model进行重构，抽象出一个tapd-sdk出来，然后本cli本质上是对这个sdk的封装和优化而已。"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 开发者通过 SDK 集成 TAPD (优先级: P1)

作为一名希望在自己的程序中访问 TAPD 数据的开发者，我希望能够引入一个独立的 tapd-sdk 包，直接调用 TAPD 的各类资源（需求、缺陷、任务、迭代等），而无需依赖整个 CLI 工具链。

**优先级原因**: 这是本次重构的核心价值——让 TAPD 能力可以被第三方程序复用，而不仅限于命令行场景。

**独立测试**: 可以编写一个独立的 Go 程序，仅引入 tapd-sdk 包，配置认证信息后调用 SDK 接口查询需求列表，验证返回数据正确且无需 CLI 相关依赖。

**验收场景**:

1. **给定** 开发者在自己的 Go 项目中引入 tapd-sdk 依赖，**当** 创建 SDK 客户端并调用查询接口时，**那么** 能够正确获取 TAPD 数据，且不引入任何 CLI 特有的依赖（如 cobra、配置文件读取等）
2. **给定** SDK 已发布，**当** 第三方项目通过标准包管理方式引入时，**那么** 能够独立编译和运行，不依赖 tapd-ai-cli 仓库

---

### 用户故事 2 - CLI 作为 SDK 的封装层正常工作 (优先级: P1)

作为现有的 tapd-ai-cli 用户，我希望 CLI 的所有命令在重构后仍能正常工作，功能和行为与重构前完全一致。

**优先级原因**: 重构不能破坏现有用户的使用体验，这是底线要求。

**独立测试**: 可以运行现有的完整测试套件，验证所有 CLI 命令（auth、story、bug、task、iteration、comment 等）的行为与重构前一致。

**验收场景**:

1. **给定** 重构完成后，**当** 运行现有 CLI 的任意命令时，**那么** 输出结果与重构前完全相同，无功能回归
2. **给定** CLI 使用 SDK 作为底层依赖，**当** SDK 返回错误时，**那么** CLI 能够正确处理并以用户友好的方式展示错误信息

---

### 用户故事 3 - SDK 支持独立认证配置 (优先级: P2)

作为 SDK 的使用者，我希望可以通过代码直接配置认证信息（Token 或 API 用户名/密码），而不依赖本地配置文件（.tapd.json），使 SDK 可以在服务端、CI/CD 等无文件系统依赖的场景中使用。

**优先级原因**: SDK 独立性的重要体现——认证管理应由调用方控制，而不是绑定到 CLI 的配置文件机制。

**独立测试**: 可以在没有 .tapd.json 文件的环境中，通过代码传入认证参数直接调用 SDK 接口，验证正常工作。

**验收场景**:

1. **给定** 没有任何本地配置文件，**当** 通过代码传入有效的认证凭据初始化 SDK 时，**那么** 能够成功调用 TAPD 接口
2. **给定** SDK 支持两种认证方式（Bearer Token 和 Basic 认证），**当** 分别使用两种方式初始化时，**那么** 均能正常访问对应权限范围内的 TAPD 资源

---

### 用户故事 4 - SDK 提供清晰的公开接口 (优先级: P2)

作为 SDK 的使用者，我希望 SDK 暴露清晰、稳定的公开 API，包含完整的数据模型定义，使我能够在不查阅源码的情况下理解如何使用各个接口。

**优先级原因**: SDK 的可用性和可维护性依赖于良好定义的公开接口边界。

**独立测试**: 可以仅通过 SDK 包的公开符号（无需查看实现）完成一个完整的 TAPD 操作（创建、查询、更新），验证接口的完备性和自解释性。

**验收场景**:

1. **给定** SDK 已发布，**当** 查看其公开接口文档时，**那么** 能够清晰了解支持的所有 TAPD 资源类型及对应操作
2. **给定** SDK 的数据模型（需求、缺陷、任务等），**当** 使用这些模型时，**那么** 字段命名和结构与 TAPD 官方 API 文档一致

---

### 边界情况

- 当 SDK 被独立引入时，不应引入 CLI 特有的依赖（命令行解析库、配置文件读取等）
- 当网络请求失败或 TAPD 返回错误状态时，SDK 应返回结构化错误，而不是直接退出进程
- 当 SDK 版本升级时，CLI 应能在不修改自身业务逻辑的前提下跟随升级

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须将 TAPD API 的数据模型（需求、缺陷、任务、迭代、评论、Wiki、附件等全部实体）从 CLI 内部迁移到独立可复用的 SDK 包中
- **FR-002**: 系统必须将 TAPD API 的 HTTP 客户端（认证、请求构造、响应解析、错误处理）从 CLI 内部迁移到 SDK 包中
- **FR-003**: SDK 必须支持通过代码参数（而非配置文件）传入认证信息，支持 Bearer Token 和 Basic 认证两种方式
- **FR-004**: SDK 必须将所有错误以结构化的错误类型返回，不得在内部终止进程
- **FR-005**: CLI 必须重构为 SDK 的消费者，通过 SDK 公开接口完成所有 TAPD 数据操作
- **FR-006**: CLI 保留对本地配置文件（.tapd.json）的读写管理，将读取的配置传递给 SDK 进行认证初始化
- **FR-007**: SDK 必须提供覆盖所有现有 CLI 功能对应的接口（需求、缺陷、任务、迭代、评论、Wiki、附件、关联关系、工时、工作流、自定义字段、工作区等）
- **FR-008**: 重构后所有现有 CLI 命令的功能和输出必须与重构前保持完全一致

### 关键实体

- **SDK 客户端**: 表示与 TAPD API 的连接会话，持有认证信息和连接配置，是调用所有接口的入口点
- **TAPD 资源模型**: 表示 TAPD 中的各类业务实体（需求 Story、缺陷 Bug、任务 Task、迭代 Iteration、评论 Comment、Wiki、附件 Attachment 等），包含完整的字段定义
- **请求参数结构**: 表示调用各接口时的输入参数，提供类型安全的参数传递方式
- **SDK 错误类型**: 表示 TAPD API 返回的各类错误（HTTP 错误、业务错误），包含状态码和错误信息

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: tapd-sdk 可作为独立包被引入，第三方 Go 项目无需依赖 CLI 相关代码即可完整使用所有 TAPD API 功能
- **SC-002**: 重构后 CLI 的全部现有测试（单元测试 + 集成测试）100% 通过，零功能回归
- **SC-003**: tapd-sdk 的直接依赖数量不超过实现 HTTP 通信和 JSON 解析所必需的最小集合（排除 CLI 特有依赖如 cobra 等）
- **SC-004**: CLI 代码中不再包含直接的 TAPD API HTTP 调用逻辑，所有数据操作均通过 SDK 接口完成
- **SC-005**: SDK 接口覆盖现有 CLI 支持的全部 TAPD 资源类型（不少于 12 种资源：story、bug、task、iteration、comment、wiki、attachment、relation、timesheet、workflow、workspace、custom_field）

## 假设

- tapd-sdk 将作为当前仓库内的子模块（子目录 + 独立 go.mod）或独立 Go 模块发布，具体发布形式由规划阶段确定
- 重构过程中保持向后兼容，不引入破坏性变更到 CLI 的外部接口（命令名称、参数、输出格式）
- SDK 的包名和导入路径将在规划阶段确定，本规范不做技术细节约束
