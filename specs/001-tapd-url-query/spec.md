# 功能规范: TAPD URL 通用查询命令

**功能分支**: `001-tapd-url-query`
**创建时间**: 2026-04-16
**状态**: 草稿（已更新：扩展至 4 种对象类型）
**输入**: 用户描述: "根据TAPD的URL获取对应内容的通用命令，支持需求(Story)、缺陷(Bug)、任务(Task)、Wiki四种类型，每种类型有多种URL格式"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 通过 Story URL 查询需求详情 (优先级: P1)

用户拥有一个 TAPD 需求的链接（可能是详情页 URL 或列表页带预览参数的 URL），希望直接通过该链接获取需求完整信息，无需手动提取 ID。

**优先级原因**: 需求是项目管理最核心的条目类型，Story URL 是最高频分享场景。

**独立测试**: 执行 `tapd url <story-url>` 后验证返回结果包含对应 Story 的标题、状态等字段。

**验收场景**:

1. **给定** 用户拥有 Story 详情页 URL（如 `https://www.tapd.cn/tapd_fe/51081496/story/detail/1151081496001028684`），**当** 执行 `tapd url <url>` 时，**那么** 系统返回该需求的完整详情。
2. **给定** 用户拥有 Story 列表页带预览参数的 URL（含 `dialog_preview_id=story_<id>`），**当** 执行 `tapd url <url>` 时，**那么** 系统从查询参数中提取 ID 并返回对应需求详情。

---

用户拥有一个 TAPD 缺陷的链接（详情页或列表页预览 URL），希望直接获取缺陷完整信息。

**优先级原因**: Bug 是另一个高频引用类型，与 Story 并列为核心场景。

**独立测试**: 执行 `tapd url <bug-url>` 后验证返回结果包含 Bug 的标题、严重程度、状态等字段。

**验收场景**:

1. **给定** 用户拥有 Bug 详情页 URL（如 `https://www.tapd.cn/tapd_fe/51081496/bug/detail/1151081496001016136`），**当** 执行 `tapd url <url>` 时，**那么** 系统返回该缺陷的完整详情。
2. **给定** 用户拥有 Bug 列表页带预览参数的 URL（含 `dialog_preview_id=bug_<id>`），**当** 执行 `tapd url <url>` 时，**那么** 系统正确解析并返回对应缺陷详情。

---

用户拥有一个 TAPD 任务的链接（详情页或任务看板预览 URL），希望直接获取任务完整信息。

**优先级原因**: Task 是与 Story/Bug 并列的核心对象，AI Agent 在处理工作分配时频繁引用。

**独立测试**: 执行 `tapd url <task-url>` 后验证返回结果包含任务的标题、状态、处理人等字段。

**验收场景**:

1. **给定** 用户拥有 Task 详情页 URL（如 `https://www.tapd.cn/tapd_fe/51081496/task/detail/1151081496001028786`），**当** 执行 `tapd url <url>` 时，**那么** 系统返回该任务的完整详情。
2. **给定** 用户拥有任务看板页带预览参数的 URL（如 `https://www.tapd.cn/51081496/prong/tasks?...&dialog_preview_id=task_<id>`），**当** 执行 `tapd url <url>` 时，**那么** 系统正确解析并返回对应任务详情。

---

### 用户故事 4 - 通过 Wiki URL 查询 Wiki 文档内容 (优先级: P2)

用户拥有一个 TAPD Wiki 文档的链接，希望直接获取 Wiki 文档内容，无需进入浏览器查看。

**优先级原因**: Wiki 是知识沉淀的重要载体，AI Agent 需要读取 Wiki 内容以获取项目背景知识。

**独立测试**: 执行 `tapd url <wiki-url>` 后验证返回结果包含 Wiki 的标题和正文内容。

**验收场景**:

---

### 用户故事 5 - 无效或不支持的 URL 处理 (优先级: P2)

用户误输入了格式错误的 URL 或不支持的 TAPD 条目类型，系统给出清晰提示。

**优先级原因**: 良好的错误处理避免用户困惑，尤其是 URL 格式复杂多样时。

**独立测试**: 传入非 TAPD URL 或不支持类型 URL，验证系统返回含可操作建议的错误消息。

**验收场景**:

1. **给定** 用户传入非 TAPD 域名的 URL，**当** 执行 `tapd url <url>` 时，**那么** 系统显示明确错误，说明该 URL 不是有效的 TAPD URL。
2. **给定** 用户传入 TAPD URL 但类型不在支持范围内，**当** 执行命令时，**那么** 系统提示当前支持的类型（Story、Bug、Task、Wiki）。

---

### 边界情况

- 当 URL 中的工作区 ID 与当前配置不一致时，系统自动使用 URL 中的工作区 ID 进行查询。
- 当用户无权访问 URL 对应的工作区或条目时，系统返回清晰的权限错误提示。
- 当 URL 格式合法但对应条目已被删除时，系统返回资源不存在的提示。
- Wiki URL 使用 `#fragment` 携带 ID，系统需正确解析 fragment 部分（而非路径或查询参数）。
- `dialog_preview_id` 参数值带有类型前缀（如 `story_`、`bug_`、`task_`），系统需去掉前缀后提取纯数字 ID。
- TAPD 前端路径可能包含 `tapd_fe` 前缀段，系统需兼容此格式。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须提供一个名为 `url` 的顶级子命令，接受一个 TAPD URL 作为参数。
- **FR-002**: 系统必须能够解析 TAPD URL，自动识别条目类型（Story、Bug、Task、Wiki）以及工作区 ID 和条目 ID。
- **FR-003**: 系统必须同时支持两种 URL 格式：`/tapd_fe/{workspace_id}/{type}/detail/{id}` 详情页格式，以及含 `dialog_preview_id={type}_{id}` 查询参数的列表页/看板页格式。
- **FR-004**: 系统必须支持 Wiki URL 的 fragment 格式（`#fragment` 中携带 ID）。
- **FR-005**: 当 URL 被识别为 Story 类型时，系统必须获取并返回该 Story 的完整详情。
- **FR-006**: 当 URL 被识别为 Bug 类型时，系统必须获取并返回该 Bug 的完整详情。
- **FR-007**: 当 URL 被识别为 Task 类型时，系统必须获取并返回该 Task 的完整详情。
- **FR-008**: 当 URL 被识别为 Wiki 类型时，系统必须获取并返回该 Wiki 文档的完整内容。
- **FR-009**: 系统必须优先使用 URL 中携带的工作区 ID 进行查询，覆盖当前配置中的工作区设置。
- **FR-010**: 当输入 URL 不是有效的 TAPD URL 时，系统必须返回清晰的错误消息。
- **FR-011**: 当 URL 对应的条目类型不在支持范围内时，系统必须提示用户当前支持的类型。
- **FR-012**: 系统必须支持与其他命令一致的输出格式（默认紧凑 JSON，`--pretty` 标志输出格式化 JSON）。

### 关键实体

- **TAPD URL**: 指向 TAPD 平台某个条目的链接，包含工作区 ID、条目类型标识和条目 ID，有多种路径格式。
- **Story（需求）**: TAPD 中的需求条目，包含标题、状态、描述、负责人、优先级等属性。
- **Bug（缺陷）**: TAPD 中的缺陷条目，包含标题、严重程度、状态、负责人、重现步骤等属性。
- **Task（任务）**: TAPD 中的任务条目，包含标题、状态、处理人、所属需求等属性。
- **Wiki（文档）**: TAPD 中的 Markdown Wiki 文档，包含标题、正文内容、创建人等属性。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 用户从拥有 TAPD URL 到通过命令行看到完整条目内容，整个操作在 10 秒内完成（网络正常情况下）。
- **SC-002**: 对于权限足够的有效 Story、Bug、Task、Wiki URL（所有格式），命令执行成功率达到 100%。
- **SC-003**: 用户无需查阅文档，仅凭 `--help` 提示即可在首次使用时正确执行该命令。
- **SC-004**: 将"通过 URL 获取条目详情"的操作步骤从两步（提取 ID → 执行 show 命令）缩减为一步。

## 假设

- TAPD URL 域名为 `www.tapd.cn` 或 `tapd.cn`，可作为 TAPD URL 的识别依据。
- `tapd_fe` 路径格式中，工作区 ID 位于 path 的第二段（`/tapd_fe/{workspace_id}/...`）。
- 直接路径格式中，工作区 ID 位于 path 的第一段（`/{workspace_id}/...`）。
- `dialog_preview_id` 参数值的前缀与条目类型对应（`story_`、`bug_`、`task_`），去掉前缀后即为纯 ID。
- Wiki URL 中条目 ID 在 URL fragment（`#` 后）中携带。
- 用户已完成认证配置，`url` 命令与其他命令共享相同认证机制。
- Wiki 查询需要新增 client 方法，调用 TAPD Markdown Wiki API。
